Tecnica di utilizzo del comando exec senza comando per aprire file descriptors ðŸ”—

Using file descriptors is more efficient (reducing processing time by about 5x). This is because the file is not opened/closed implicitly between operations.

#!/usr/bin/env bash

function readfiles()
{
local FD1=7
local FD2=8
local file1=$1
local file2=$2
local count1=0
local count2=0
local eof1=0
local eof2=0
local data1
local data2
# Open files. Use file descriptors in the range 3-9. File descriptors below 3 are used for standard input, output,
# and error, the ones above 9 may be used by the shell internally.
#
# As a reference, the man page describes what exec is doing:
# exec [-cl] [-a name] [command [arguments]]
# If command is not specified, any redirections take effect in the current shell, and the return status is 0.
# If there is a redirection error, the return status is 1.
exec 7<$file1
exec 8<$file2
# E' possibile parametrizzare i numeri dei file descriptor con questo trucco:
# eval exec "$FD1<$file1"
while [[ $eof1 -eq 0 || $eof2 -eq 0 ]]
do
if read data1 <&$FD1; then # Notice the input redirection to read uses another special form that includes the
# ampersand (&) to specify that what follows is a file descriptor and not a file name.
let count1++
printf "%s, line %d: %s\n" $file1 $count1 "$data1"
else
eof1=1
fi
if read data2 <&$FD2; then
let count2++
printf "%s, line %d: %s\n" $file2 $count2 "$data2"
else
eof2=1
fi
done
}
echo "Reading file1 and file2"
readfiles file1 file2
echo "Reading file3 and file4"
readfiles file3 file4

Closing File Descriptors ðŸ”—

To be safe you could do the following to close the files:

exec 7</dev/null
exec 8</dev/null

There is an explicit syntax for closing a file descriptor. If you want to close descriptor 7:

exec 7>&-

I recommend avoiding this with descriptors 0-2, since many programs will behave erratically if run with these descriptors closed.
