--soft permette di preservare lo stato della working directory e le modifiche aggiunte all'index non ancora committate,
utilizzando --hard le modifiche locali vengono perdute, sia index che working directory.

Con il reset soft si torna indietro annullando i commit ma mantenendo lo stato della working directory e dell'index, in questo modo non si perde alcuno stato delle modifiche e si puÃ² decidere cosa fare nell'index e nella working directory in modo selettivo.

Con ~1 si indica il primo ancestor del commit HEAD.

$ git reset --soft HEAD~
$ git reset --soft HEAD~1

la cosa funzione anche con un numero di ancestors superiore ad 1.

Ora che l'HEAD Ã¨ stato riportato indietro, i commit seguenti sono diventati orfani e saranno eliminati, tuttavia grazie al --soft la working directory contiene ancora i file allo stato in qui erano prima del reset, si puÃ² decidere di effettuare uno squash, facendo subito commit e dunque di fatto sostituendo tutti i commit orfani con un unico nuovo commit ,oppure fare altro.

Significato del caret ^

git rev-parse --help

<rev>^[<n>], e.g. HEAD^, v1.5.1^0
       A suffix ^ to a revision parameter means the first parent of that commit object.  ^<n> means the <n>th parent (i.e.  <rev>^ is equivalent to <rev>^1). As a special rule,
       <rev>^0 means the commit itself and is used when <rev> is the object name of a tag object that refers to a commit object.
