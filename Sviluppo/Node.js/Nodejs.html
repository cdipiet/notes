Dato un numero di versione MAJOR.MINOR.PATCH, incrementate la:

  • versione MAJOR quando modificate lâ€™API in modo non retrocompatibile,
  • versione MINOR quando aggiungete funzionalitÃ  in modo retrocompatibile
  • versione PATCH quando correggete bug in modo retrocompatibile.

include everything greater than a particular version in the same major range use the carat (aka hat) symbol, ^:

^2.2.1

include everything greater than a particular version in the same minor range use the tilde symbol, ~:

~2.2.0

specify a range of stable versions use >, <, =, >= or <= for comparisons, or - to specify an inclusive range:

  • >2.1
  • 1.0.0 - 1.2.0

Note: there must be spaces on either side of hyphens

include multiple sets of versions use || to combine:

^2 <2.2 || > 2.3

Package versions ðŸ”—

You have seen in the description above version numbers like these: ~3.0.0 or ^0.13.0. What do they mean, and which other version specifiers can you use?

That symbol specifies which updates your package accepts, from that dependency.

Given that using semver (semantic versioning) all versions have 3 digits, the first being the major release, the second the minor release and the third is the patch release, you have these rules:

  • ~: if you write ~0.13.0, you want to only update patch releases: 0.13.1 is ok, but 0.14.0 is not.
  • ^: if you write ^0.13.0, you want to update patch and minor releases: 0.13.1, 0.14.0 and so on.
  • *: if you write *, that means you accept all updates, including major version upgrades.
  • >: you accept any version higher than the one you specify
  • >=: you accept any version equal to or higher than the one you specify
  • <=: you accept any version equal or lower to the one you specify
  • <: you accept any version lower to the one you specify

There are other rules, too:

  • no symbol: you accept only that specific version you specify
  • latest: you want to use the latest version available

and you can combine most of the above in ranges, like this: 1.0.0 || >=1.1.0 <1.2.0, to either use 1.0.0 or one release from 1.1.0 up, but lower than 1.2.0.

Tecnica per trasformare un .bat in un eseguibile Node.js

%~f0 Ã¨ il nome del presente file con il path completo
L'intestazione rem=` trasforma il presente .bat in un file eseguibile Node.js.
Di seguito la sequenza delle operazioni:

  1. la prima riga rem=` viene ignorata da cmd.exe in quanto un commento
  2. la seconda riga node "%~f0" %* richiama questo stesso file ma con l'interprete node
  3. node interpreta la riga rem=` fino alla riga del successivo ` come l'assegnazione alla variabile rem di una stringa template multiriga, viene cosÃ¬ ignorato da 'node' l'insieme delle istruzioni eseguibili del .bat
    CiÃ² Ã¨ possibile perchÃ¨ non essendo in strict mode l'assegnazione rem=` non necessita di let o var
  4. node continua ad intepretare a questo punto tutte le istruzioni javascript seguenti il secondo `
rem=`
rem node --inspect-brk "%~f0" %*
rem set PATH=F:\app\node;%PATH%
net use \\192.168.1.2 /user:YYYYYY XXXXXXX
node "%~f0" %*
pause :: blocco del terminale a scopo di visualizzazione
exit /b %errorlevel%	:: Uscita dal batch file
`
/*
	https://nodejs.org/api/debugger.html
	chrome://inspect/
	usare 'node --inspect-brk' per debug con client chrome
	usare 'debugger;' nel codice .js per inserire un breakpoint
	usare 'node inspect' per effettuare il debug da commandline
*/

const os = require('os');
const {join:_} = require('path');

/**
 * Modo per importare un modulo ESM da un modulo CommonJS
 */
async function importESM_from_CommonJS() {
    const {sync, user, home} = await import (_('file:///D:', 'workspaces', 'Nodejs', 'Applications', 'DirSync', 'index.js') );
    const LOGDIR = _('D:', 'Dati', 'Profili', user, 'log');
    // https://stackoverflow.com/questions/61416857/why-cant-string-raw-end-with-a-backslash
    const SOURCE_DRIVE        = '\\\\192.168.1.2';
    const SOURCE_DIR_USER     = _(SOURCE_DRIVE, '$USER');
    const SOURCE_DIR_WS       = _(SOURCE_DIR_USER, 'workspaces');
    const DEST_DRIVE          = 'D:';
    const DEST_DIR_WS         = _(DEST_DRIVE, 'workspaces');
    const DEST_DIR_USER       = _(DEST_DRIVE, 'Dati', 'Profili', user);

    sync([
        [_(SOURCE_DIR_USER, 'Documenti', 'per'),  _(DEST_DIR_USER, 'Documents', 'per')],
        [_(SOURCE_DIR_WS, 'Python'),              _(DEST_DIR_WS, 'Python'), '/XD "target" /XD "node_modules" /XF "*.pyc"'],
        [_(SOURCE_DIR_WS, 'Nodejs'), _(DEST_DIR_WS, 'Nodejs'), '/XD "node_modules"'],
        //[_(SOURCE_DIR_WS, 'Nodejs', 'Applications', 'Notes'), _(DEST_DIR_WS, 'Nodejs', 'Applications', 'Notes'), '/XD "node_modules"'],
        //[_(SOURCE_DIR_WS, 'Nodejs', 'Applications'), _(DEST_DIR_WS, 'Nodejs', 'Applications'), '/XD "node_modules"'],
        [_(SOURCE_DIR_WS, 'configuration', 'Manjaro_Install'), _(DEST_DIR_WS, 'Manjaro_Install')],
        [_(SOURCE_DIR_USER, 'Musica', 'Music'),   _(DEST_DIR_USER, 'Music')]
    ], LOGDIR);
}
